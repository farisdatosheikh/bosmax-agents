# BOSMAX VIDEO SOURCE MODE CONTRACT v1

```
Status:       CANONICAL — visual intake routing authority for BOSMAX video asset handling
Version:      v1.0
Date:         2026-06-09
Authority:    BOSMAX repo truth
Scope:        Registered product fast intake, on-the-fly full intake, asset role binding,
              and source-mode declaration before storyboard or prompt generation
```

---

## 1. Why This Contract Exists

The BOSMAX video lane already has visual-first enforcement, product truth gates, avatar
source rules, and on-the-fly routing. What was missing was one explicit contract that forces
the system to declare how uploaded visuals are being used before planning storyboards,
prompts, or future batch variants.

This contract exists to stop five recurring routing failures:
- registered products being re-analysed as if they are unknown on every intake
- avatar uploads being silently ignored in favor of registry personas
- style/background references overriding product truth
- ready-frame inputs being treated like ordinary multi-variation batch defaults
- on-the-fly products being upgraded into reusable truth without approval

Notion is downstream only. This contract governs repo truth, not UI behavior.

---

## 2. Product Truth Tiers

### REGISTERED_PRODUCT_FAST_PATH

Use when the product is already approved in `products/*.yaml` or the registered library path
and the uploaded product image roughly matches the expected product identity.

Rules:
- Product truth source = registry / approved product YAML
- Full product recognition = skipped
- Fast visual intake = required
- Image role classification = required
- Mismatch check = required
- Packaging conflict check = required
- Crop / blur / label obstruction check = required
- Source mode decision = required
- Uploaded visuals may challenge a mismatch, but they do not re-author claims from scratch

### ON_THE_FLY_FULL_SCAN_PATH

Use when the product is unknown, unregistered, or cannot safely inherit approved product truth.

Rules:
- Product truth source = current uploaded intake only
- Full visual intake = required
- Extract category, visible brand, label text, packaging, dominant color, material, and
  claim-risk signals from the current session
- Registry writeback = forbidden
- Session scope = current session only
- Risky inputs must fail closed into review-only posture, not silent generation

---

## 3. Source Modes

### REFERENCE_SET_MODE

Use when the user uploads multiple references that each serve separate roles.

Typical shape:
- product image
- avatar image
- style/background image

Requirements:
- asset binding table required
- product truth priority depends on registered vs on-the-fly path
- uploaded avatar image overrides registry avatar
- style image may influence environment, lighting, or mood only
- style image must not override product truth

### HYBRID_PRODUCT_ANCHOR_MODE

This is the default recommended source mode for registered product video batch planning.

Use when:
- the user uploads a product image only
- or the user selects a registered product with an approved product reference

Requirements:
- product acts as the visual anchor
- avatar comes from approved avatar pool or approved session avatar
- scene/style is generated by prompt
- product lock required
- avatar source resolution required
- scene variation plan required
- execution variant plan required for future batch work
- registered products skip full product recognition

### READY_FRAME_MODE

Use when the user uploads a single completed frame already containing product plus
avatar/subject plus scene/style.

Best use:
- animate / continue approved frame
- maintain a specific approved visual composition

Requirements:
- source frame scan required
- motion-delta-only instruction required
- no product redesign
- not recommended as the default for a 10-video batch because visual diversity collapses too fast

### ON_THE_FLY_FULL_SCAN_MODE

Use when the product is unknown or not registered.

Requirements:
- full product truth extraction required
- session-only posture
- registry writeback forbidden
- risky products route to review-only instead of session-only generation

## 5. Asset Role Classification

Every uploaded visual must be classified before storyboard or prompt generation.

| Asset role | Meaning |
|------------|---------|
| `PRODUCT_REFERENCE` | Product-only or product-dominant reference used to lock product truth |
| `AVATAR_REFERENCE` | Human/avatar reference used to lock identity, wardrobe, and pose DNA |
| `STYLE_SCENE_REFERENCE` | Environment, lighting, framing, or texture reference only |
| `READY_FRAME_PRODUCT_AVATAR` | Complete ready frame containing product + human + scene |
| `READY_FRAME_PRODUCT_ONLY` | Nearly finished frame containing product scene truth but no usable avatar |
| `UNKNOWN_OR_AMBIGUOUS` | Role cannot be safely inferred and must be clarified or blocked |

Binding law:
- `AVATAR_REFERENCE` overrides registry avatar when uploaded
- `STYLE_SCENE_REFERENCE` never overrides product truth
- `UNKNOWN_OR_AMBIGUOUS` may not silently pass into prompt generation

---

## 6. Registered Product Fast Intake

When `REGISTERED_PRODUCT_FAST_PATH` is active:
- product truth source = registry
- full product recognition is skipped
- system still scans:
  - asset role
  - source mode
  - visual mismatch
  - packaging conflict
  - crop / blur / label obstruction
- system must not re-derive product claims from the uploaded image
- if uploaded visual contradicts the registered product strongly, hold and escalate instead of drifting

---

## 7. On-The-Fly Full Intake

When `ON_THE_FLY_FULL_SCAN_PATH` is active:
- full product extraction is mandatory
- extract only from current intake, not from registry inheritance
- output remains session-only
- registry writeback is forbidden
- if claim-risk or category-risk is high, block into review-only posture

---

## 8. Batch Implications

- Default source mode for a registered video batch = `HYBRID_PRODUCT_ANCHOR_MODE`
- Default future video batch execution posture should later resolve to `VIDEO_EXECUTION_VARIANT_BATCH`
- This contract does not open batch production pages here
- This contract does not approve page 04 / 06 execution
- `READY_FRAME_MODE` is allowed for continuation work, but it is not the default for 10-video variation output

---

## 9. Rejection / Block Rules

Block or hold if any of the following occurs:
- uploaded assets exist but `source_mode` is not declared
- uploaded assets exist but no asset role map exists
- registered product image mismatch is ignored
- on-the-fly product is treated as registered truth
- style image overrides product truth
- uploaded avatar is ignored in favor of a registry avatar
- ready frame is treated as default 10-video batch input without review
- ambiguous asset role is passed downstream without clarification
- registry writeback is attempted for an on-the-fly session

## 10. Examples

### BOSMAX product-only upload

Input:
- registered BOSMAX product selected
- one product image uploaded

Result:
- `REGISTERED_PRODUCT_FAST_PATH`
- `HYBRID_PRODUCT_ANCHOR_MODE`
- registry truth kept
- fast mismatch / packaging / crop scan required

### Minyak Warisan baby tummy routine cue

Input:
- registered Minyak Warisan product
- product image only
- user asks for baby tummy routine cue

Result:
- `REGISTERED_PRODUCT_FAST_PATH`
- `HYBRID_PRODUCT_ANCHOR_MODE`
- product truth from registry
- avatar resolved from approved pool or uploaded session avatar
- scene routine may vary, product truth may not

### Multi-image product + avatar + style

Input:
- product image
- separate avatar image
- separate room/background/style image

Result:
- `REFERENCE_SET_MODE`
- asset binding table required
- product truth follows registered/on-the-fly path
- uploaded avatar overrides registry persona
- style image influences scene only

### Ready frame avatar + product

Input:
- one completed frame already containing product + human + scene

Result:
- `READY_FRAME_MODE`
- source frame scan required
- motion delta only
- no product redesign
- hold before any large-scale variation batch

### Unknown product

Input:
- product image uploaded
- product not found in registry

Result:
- `ON_THE_FLY_FULL_SCAN_PATH`
- `ON_THE_FLY_FULL_SCAN_MODE`
- full extraction required
- session-only output
- no registry writeback
